# Problem: `import.meta.url` breaks when Netlify's nft bundles the UD catch-all

## Summary

When deploying a Cedar Universal Deploy (UD) project to Netlify, the catch-all
server function (`server.mjs`, generated by `@netlify/vite-plugin`) fails
because **Netlify's nft bundler inlines all imported modules into a single
bundle**, which means `import.meta.url` inside the inlined code points to the
**entry point** (`server.mjs`) instead of the **original file**. This breaks
relative path resolution in dynamically imported modules.

## Affected Code

All three handler patterns in Cedar's UD pipeline use relative URLs based on
`import.meta.url`:

### 1. `createFunctionHandler` (lazy dynamic import)

Generated handler file at `api/dist/ud/.api_functions_foo-handler.js`:

```js
import { createFunctionHandler } from '@cedarjs/vite/ud-handlers/function'
const foo = createFunctionHandler({
  distUrl: new URL('./../functions/foo.js', import.meta.url).href,
})
export { foo as default }
```

### 2. `createGraphQLHandler` (lazy dynamic import)

Generated handler file at `api/dist/ud/.api_functions_graphql-handler.js`:

```js
import { createGraphQLHandler } from '@cedarjs/vite/ud-handlers/graphql'
const graphql = createGraphQLHandler({
  distUrl: new URL('./../functions/graphql.js', import.meta.url).href,
})
export { graphql as default }
```

### 3. `createFunctionHandlerFromModule` (inline module)

Also uses `import.meta.url` indirectly through `createCedarFetchable` →
`buildCedarContext`, which triggers the `@cedarjs/api` self-resolution pattern
(`require.resolve("@cedarjs/api")`) that also doesn't survive nft bundling.

Both `createFunctionHandler` and `createGraphQLHandler` do
`await import(options.distUrl)` to lazily load the transpiled function module at
runtime.

## What happens (step by step)

### Development / local

1. `server.mjs` statically imports `api/dist/ud/index.js` (the UD router)
2. UD router lazily `import()`s `./.api_functions_graphql-handler.js`
3. `import.meta.url` inside the handler →
   `file:///.../api/dist/ud/.api_functions_graphql-handler.js`
4. `new URL("./../functions/graphql.js", import.meta.url)` → resolves correctly
   to `api/dist/functions/graphql.js`
5. `import(options.distUrl)` loads the transpiled function — works ✅

### Deployed to Netlify

1. Netlify's nft bundler bundles `server.mjs` and **inlines all its
   dependencies**, including the UD router and all handler files, into a single
   `___netlify-bootstrap.mjs` bundle
2. The inlined code looks like this (simplified from actual output):

```js
// All inlined into ___netlify-bootstrap.mjs
var graphql
var init_api_functions_graphql_handler = __esm({
  'api/dist/ud/.api_functions_graphql-handler.js'() {
    graphql = createGraphQLHandler({
      distUrl: new URL('./../functions/graphql.js', import.meta.url).href,
    })
  },
})
```

3. `import.meta.url` inside the `__esm` wrapper →
   `file:///var/task/web/.netlify/v1/functions/server.mjs` (the entry point,
   **not** the original handler file)
4. `new URL("./../functions/graphql.js", import.meta.url)` → resolves to
   `/var/task/web/.netlify/v1/functions/graphql.js` (wrong — doesn't exist)
5. `import(options.distUrl)` fails with:

```
Error: Cannot find module '/var/task/web/.netlify/v1/functions/graphql.js'
       imported from /var/task/node_modules/@cedarjs/vite/dist/ud-handlers/graphql.js
```

## Root cause

nft bundles the entire dependency graph into a single file. Any code that uses
`import.meta.url` to compute relative paths will get the **bundled entry
point's** URL, not the original file's URL. This is a fundamental limitation of
bundling — `import.meta.url` is a runtime value that reflects the module's
actual location, and after bundling, there's only one "module" (the bundle).

Cedar's UD pipeline assumes the handler files will be deployed as separate files
at `api/dist/ud/` and `api/dist/functions/`, with the catch-all doing runtime
lazy `import()`s between them. nft defeats this by inlining everything.

## What we tried

1. **`[functions.server] included_files`** — Including `api/dist/ud/**` and
   `api/dist/functions/**` in the function zip. This puts the files at the
   correct paths in the zip, but nft still inlines everything into the bootstrap
   bundle, so the files exist but aren't used.

2. **Deploying functions individually** (via `[functions] directory`) — This
   works for simple functions (like a "foo" endpoint) but triggers a CJS/ESM
   conflict for the graphql function (see separate issue).

3. **Patching the handler files** — Replacing the `createFunctionHandler`
   wrapper with a plain `{ fetch: fn }` object. This bypasses the dynamic import
   entirely, but it only works for functions with zero Cedar dependencies.

## Potential solutions

### A. Avoid bundling for the catch-all function

Netlify Functions v2 supports opting out of nft bundling by setting
`"bundler": "none"` or configuring the function to use a different bundler. If
the catch-all function is deployed without bundling, the original ESM `import()`
calls survive and `import.meta.url` reflects the actual file locations. However,
this might break other things since `@netlify/vite-plugin` generates the
catch-all and assumes nft is used.

### B. Compute absolute paths at build time

Instead of computing `distUrl` as a relative URL at runtime using
`import.meta.url`, make the Vite plugin that generates the handler files compute
**absolute file:// URLs** at build time. This decouples the runtime resolution
from `import.meta.url`:

```js
// Instead of:
distUrl: new URL('./../functions/foo.js', import.meta.url).href

// Use an absolute path computed during build:
distUrl: 'file:///var/task/api/dist/functions/foo.js'
```

This is fragile — the absolute path depends on the deployed environment.

### C. Make the handler wrapper resolve imports differently

Instead of lazy `import(options.distUrl)`, have the handler wrapper accept the
already-imported module, similar to `createFunctionHandlerFromModule`. The
handler generator would do the import eagerly during its initialization, so the
dynamic import resolution happens at module evaluation time, not inside an
`__esm` wrapper:

```js
// Generated handler (evaluated eagerly, before nft inlines it? No — nft
// inlines everything so even eager imports get bundled)
```

This doesn't actually help because nft inlines eagerly too.

### D. Expose the function modules differently

Bundle the api functions into the catch-all as static imports rather than
dynamic lazy imports. This means the nft bundler can trace them and inline them
correctly. The downside: the function bundle gets bigger (all functions loaded
eagerly) and `createGraphQLHandler` would need to not use dynamic `import()`.

### E. Fundamentially: change the output format

If Cedar emitted single-file bundles for each function (no relative imports, no
directory structure), nft wouldn't need to resolve relative paths and
`import.meta.url` wouldn't matter. This is essentially what `esbuild --bundle`
produces.

## The graphql standalone function CJS/ESM issue

There's a second, related issue when deploying `api/dist/functions/graphql.js`
as a standalone (non-catch-all) function. The graphql module uses ESM `import`
syntax. When nft packages it, it creates a wrapper entry point:

```js
module.exports = require('./api/dist/functions/graphql.js')
```

But the root `package.json` has `"type": "module"`, so `.js` files are treated
as ESM and `module` is undefined. The nft wrapper uses CJS `module.exports` in
an ESM context, causing:

```
ReferenceError: module is not defined in ES module scope
```

This suggests nft doesn't fully respect `"type": "module"` when generating its
entry point wrappers.
