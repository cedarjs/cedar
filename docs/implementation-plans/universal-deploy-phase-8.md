# Netlify Deployment and the nft Bundling Problem

## Background

When deploying a Cedar Universal Deploy (UD) project to Netlify, the catch-all
server function (`server.mjs`, generated by `@netlify/vite-plugin`) is processed
by Netlify's [nft](https://github.com/vercel/nft) bundler before deployment. nft
traces the full dependency graph of the entry point and inlines all imported
modules into a single `___netlify-bootstrap.mjs` bundle.

The original UD handler design generated virtual modules like this:

```js
// api/dist/ud/.api_functions_graphql-handler.js
import { createGraphQLHandler } from '@cedarjs/vite/ud-handlers/graphql'
const graphql = createGraphQLHandler({
  distUrl: new URL('./../functions/graphql.js', import.meta.url).href,
})
export { graphql as default }
```

This worked locally because `import.meta.url` resolved to the handler file's
actual path on disk. After nft inlines everything into a single bundle however,
`import.meta.url` points at the bundle entry (`server.mjs`) instead, so the
relative URL resolves to the wrong location and the runtime `import()` fails.

## What We Tried

### 1. `included_files` in `netlify.toml`

Adding `[functions.server] included_files` to include `api/dist/ud/**` and
`api/dist/functions/**` in the function zip puts the files at the correct paths,
but nft still inlines everything into the bootstrap bundle — the files exist in
the zip but are never actually loaded.

### 2. Static imports with `import * as fnModule from relPath`

Replacing the lazy `import(distUrl)` with a static top-level import eliminates
the `import.meta.url` problem, since static imports are resolved by the module
system rather than computed at runtime. However, this caused the Vite/Rollup
build to fail: the `api/dist/functions/` files are produced by esbuild in a
separate prior step, and Rollup tries to resolve static imports at build time —
before those files exist — producing a `"Could not resolve"` error.

### 3. Dynamic `import('file:///absolute/path')`

Using an absolute `file://` URL string literal in a dynamic `import()` sidesteps
both Rollup's build-time resolution (Rollup doesn't trace `import()` with a
non-analysable string) and the `import.meta.url` problem (no relative path
computation). However, this approach has two fatal flaws for Netlify deployment:

- **nft won't find the files.** nft can't statically trace an opaque `file://`
  URL string, so `api/dist/functions/*.js` files are never included in the
  deployed zip.
- **The absolute path is wrong at runtime.** The path is baked in at build time
  (e.g. `/home/runner/work/...`), but Netlify Lambda runs from `/var/task/`. The
  import would always fail with a file-not-found error.

This approach fixes the CI test suite (where build and serve happen on the same
machine with a stable path) but does not work for real Netlify deployments.

### 4. How the non-UD Netlify path works (the insight)

Looking at how Cedar's existing non-UD Netlify deployment works was the key
insight. The non-UD path sets `functions = "api/dist/functions"` in
`netlify.toml`, and Netlify deploys each `*.js` file there as its own
independent Lambda function. nft runs on each file individually and traces all
relative imports itself, producing a self-contained zip per function.

Crucially, esbuild in the non-UD path runs with `bundle: false` — it only
transpiles TypeScript. **nft does the actual bundling.** This works because each
function file's imports are relative paths that nft can statically trace.

The UD path is fundamentally different: a single catch-all function must
dispatch to all API functions at runtime, which is why cross-file imports
existed in the first place.

## The Solution: Pre-bundle Each Function with esbuild

The solution mirrors what nft does for the non-UD path, but moves it into the
Cedar build step. During `buildUDApiServer()`, the Vite plugin's `load` hook now
calls `bundleDistFile()` for each function's virtual module. This runs esbuild
with `bundle: true, packages: 'external'` on `api/dist/functions/*.js` and
inlines the resulting code directly into the virtual module.

This means:

- **No `import.meta.url`** — the function code is inlined as a string, not
  referenced via a file path.
- **No dynamic `import()` of user code** — no file paths of any kind appear in
  the built output for user functions.
- **No build-time Rollup resolution failure** — `bundleDistFile()` runs inside
  the async `load` hook, which fires after the prior esbuild API build step has
  produced `api/dist/functions/*.js`.
- **nft only sees `node_modules` imports** — all relative imports are already
  gone by the time nft processes the catch-all. It traces `node_modules`
  dependencies identically to how it does for the non-UD path.

The esbuild output uses `format: 'esm'` with the trailing `export { ... }` block
stripped, so the bundled code becomes a set of plain `var`/`function`
declarations that the surrounding wrapper code can reference directly without
polluting the outer module's exports.

## Provider Compatibility

The pre-bundling approach in `bundleDistFile()` is provider-agnostic. Here is
why it works across the providers Cedar targets:

### Netlify and Vercel (nft-based)

Both use nft to trace the dependency graph of the deployed function. With user
code pre-bundled, nft only ever sees `node_modules` imports in the catch-all
output — no relative paths, no `import.meta.url`. nft traces `node_modules`
dependencies normally and includes them in the zip. ✅

### Node / bare-metal / Bun / Deno

No deployment bundler is involved at all. The catch-all output is loaded
directly from disk by the runtime. Relative imports are already gone;
`node_modules` imports resolve normally from the file system. ✅

### Cloudflare Workers (future)

Cloudflare's V8 isolate runtime has no filesystem and no `node_modules` at
runtime — everything must end up in a single bundle. This sounds like
`packages: 'external'` would be wrong, but it isn't a problem:

`bundleDistFile()` only handles inlining the _user's_ relative imports
(services, lib, directives, etc.). The remaining `@cedarjs/api/runtime`,
`@cedarjs/graphql-server`, etc. imports in the virtual module are standard
Rollup/Vite ESM imports — they are processed by the Vite build itself, not by
`bundleDistFile`. The Cloudflare adapter (`@cloudflare/vite-plugin`) then takes
the Vite SSR output and performs its own final bundling pass that inlines all
`node_modules` dependencies into a single Worker-compatible file.

So `bundleDistFile`'s `packages: 'external'` is correct for Cloudflare too: it
tells esbuild to leave `node_modules` as bare import specifiers, which Rollup
(and ultimately the Cloudflare adapter) resolves in its own pipeline.

The one real Cloudflare caveat is `platform: 'node'` and `target: 'node24'` in
`bundleDistFile`. These affect how esbuild resolves Node.js built-ins in the
user's function code. For Cloudflare Workers, Cedar functions would need to
avoid Node.js APIs regardless (that is a runtime compatibility concern, not a
bundling concern), so this setting does not introduce any new limitation.

### Summary

`bundleDistFile()` solves exactly one problem — eliminating relative imports
from user function code — and leaves everything else to each provider's own
pipeline. That clean separation of concerns is what makes the approach
provider-agnostic.

# Building

## Non-UD path (default Netlify)

1. **`buildApi()`** — esbuild with `bundle: false`. Transpiles `api/src/**/*.ts`
   → `api/dist/**/*.js`. Each file is transpiled individually; imports between
   them are preserved as-is.
2. **`buildCedarApp()`** (Vite) — builds the web client only.
3. Netlify deploys `api/dist/functions/*.js` **directly as individual Lambda
   functions** via `functions = "api/dist/functions"` in `netlify.toml`. Each
   function file imports from `node_modules` and from sibling `api/dist/` files
   (services, lib, etc.). Netlify's nft runs on each function file individually
   and traces all those relative imports, bundling them into a self-contained
   zip per function.

So nft is doing the _actual_ bundling in the non-UD path. esbuild only
transpiles (TS → JS); nft is what makes each function self-contained for
deployment.

## UD path (with `--ud`)

1. **`buildCedarApp()`** — esbuild `bundle: false` on the API side (same as
   above), plus Vite for the web client.
2. **`buildUDApiServer()`** — Vite/Rollup builds the UD catch-all router. The
   `load` hook for each function's virtual module now calls `bundleDistFile()`,
   which runs esbuild `bundle: true, packages: 'external'` on
   `api/dist/functions/*.js`.
3. The catch-all `index.js` has all function logic inlined. Netlify's nft runs
   on this single file and only needs to trace `node_modules` imports — no
   relative imports remain.

## The key difference

In the non-UD path, **nft does the bundling**. It's given unbundled files with
relative imports and traces the whole graph itself.

In the UD path, **esbuild pre-bundles each function** (in step 2) before nft
ever sees it. By the time nft runs, all relative imports are already gone — it
only sees `node_modules` specifiers, which it handles identically to the non-UD
case.

The redundancy (esbuild transpiling in step 1, then esbuild bundling the same
output in step 2) exists because the first esbuild pass is broad — it processes
all of `api/src/` uniformly with `bundle: false`. The UD-specific bundling in
`bundleDistFile` is a focused second pass only on the function entry points. The
cleanest long-term fix would be to have `buildApi` use
`bundle: true, packages: 'external'` for `api/src/functions/*.ts` entries
specifically, which is how the non-UD path ends up (via nft) anyway — but that's
a separate and more impactful change.
